# Threads Platform Reference

Threads formatting mirrors Twitter/X with the following deltas.
Load `platform_constrains.json` as the base, then apply these overrides.

---

## Key Differences from Twitter

### Character & Length
- **No hard character limit** per post (vs Twitter's 280)
- This does NOT mean write longer — use the extra room for a slightly more expansive single
  post when the idea needs it, but keep the scannability standard identical
- Still max 1–2 sentences per line chunk. Still short paragraphs.

### Thread Format
- Threads posts are individual replies in a chain, not numbered tweets
- **No (1/N) numbering** — don't use "(1/7)" style markers
- **No 🧵 opener** — the chain format is self-evident on Threads
- Each post in the chain should still stand reasonably alone
- The chain opener is still your highest-attention asset — same hook standards apply

### CTAs
- **No "Comment KEYWORD" mechanic** — Threads doesn't support the same keyword-DM automation
- Replace keyword CTAs with:
  - "Reply with [KEYWORD] and I'll drop the link" (manual reply)
  - "Like this if you want the full breakdown — I'll share it next"
  - "Follow me for more frameworks like this"
  - Direct link sharing is more natural on Threads than Twitter

### Links
- Links are clickable inline on Threads (no URL suppression like early Twitter)
- Can place links mid-post if natural, not just at the end
- Still avoid cluttering posts with multiple links

### Hashtags
- Same rule as Twitter: **no hashtags** for Subho's content

### Tone Calibration
- Threads skews slightly more conversational and less "performance-y" than Twitter
- The high-pressure growth hook style (Twitter-native) can feel slightly aggressive on Threads
- For Threads-specific content, soften the hook by ~10%:
  - Twitter: "Most developers waste 10 hours/week. Here's why:"
  - Threads: "I spent 10 hours on something that now takes 8 minutes. Here's what changed:"
- Keep the same substance; adjust the delivery slightly more story-forward

### Media
- Images and video perform similarly to Twitter
- Workflow diagrams and output screenshots are just as effective
- GIFs work but feel slightly less native — use sparingly

## When to Post on Threads vs Twitter

Post on **both** by default unless the user specifies otherwise.

Post on **Twitter only** when:
- The content is highly technical and developer-specific
- The CTA uses a keyword-trigger automation
- The topic is fast-moving/reactive (Twitter moves faster)

Post on **Threads only** when:
- The content is more personal/conversational
- You want to test a softer version of a Growth hook
- The user is running a Threads-specific engagement experiment

---

## Adapting a Twitter Post for Threads

When converting a Twitter post to Threads:
1. Remove (1/N) numbering from thread posts
2. Remove 🧵 opener
3. Change "Comment KEYWORD" CTA to "Reply with KEYWORD" or a follow CTA
4. Soften any extremely aggressive hooks by ~10% (story-forward phrasing)
5. Links can move mid-post if it reads more naturally
6. Everything else — hook quality, scannability, brand voice, no hashtags — stays identical
